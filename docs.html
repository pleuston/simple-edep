Guide

This page walks through the editing workflow. simple-edep is a simple rewrite of the EDEP edition tool for serverless deployment, so the steps mirror EDEP's, with one difference: there is no backend — the XML is built in your browser and saved to GitHub with your own token.

For developers and contributors, see the Developer guide →

1 · Access

Reading is open to everyone: you can browse Inscriptions, Geodata and People without an account. To create or save a record you sign in once with a GitHub personal access token (public_repo scope, or repo for a private corpus). The token is kept only in your browser and is sent solely to the GitHub API.

2 · Create or open a record

Start a fresh record with + New inscription, or open an existing one from Inscriptions with Edit — its metadata and transcription load back into the form. Give each record a short file name (e.g. rom_001); it becomes records/<file>.xml.

3 · Describe the object and its context

Fill the metadata sections from the top down. The right-hand pane shows the EpiDoc XML live and flags whether it is well-formed.

  • Identity & IDs — title, editor, and cross-references (EDH, EDCS, EDR, Trismegistos, PHI, CIL → idno).
  • Object & support — object type and material (with EAGLE / Getty AAT URIs), dimensions, letter heights, condition.
  • Layout & lettering — columns, lines, script (handNote).
  • Text type & dating — EDH text type and origDate (see step 5).
  • Find spot — modern country/region/settlement, coordinates (used by Geodata), provenance, repository.

4 · Transcribe the text

The transcription pane is a live Leiden+ → EpiDoc editor (jinn-codemirror, the same Leiden engine EDEP uses). Type with Leiden conventions and the EpiDoc XML is produced automatically. You can also paste raw text from EDCS or PHI and use the Import buttons to convert it.

You type (Leiden+)EpiDoc XMLMeaning
[Imp]<supplied reason="lost">Imp</supplied>Lost text, restored
(C(aesar))<expan><abbr>C</abbr><ex>aesar</ex></expan>Abbreviation expanded
〚abc〛<del rend="erasure">abc</del>Erasure (rasura)
ạ (underdot)<unclear>a</unclear>Unclear letter
[.5]<gap reason="lost" quantity="5" unit="character"/>Lacuna, 5 lost letters
1. …<lb n="1"/>…Line number

Use the toolbar buttons for the common cases, and ↩ to (re)number lines. The Fullscreen button (⤢) opens the editor across the whole window for distraction-free work; press Esc to leave it.

Tip — lacunae inside restorations. A lost-letter gap that belongs outside a restoration can end up nested inside it: typing [- - - t]ribuno and then turning the dashes into a gap yields the wrong [[- - -] t]ribuno. Keep the gap element outside the supplied so it renders as [- - - t]ribuno — edit in the XML pane if needed.

5 · Names, places and dates

Add the apparatus, translation, commentary and bibliography in their own sections. For dating, give origDate a year or range (@when / @notBefore / @notAfter), a dating method, and the date as expressed on the stone. Places carry a Pleiades / Trismegistos Geo URI and the Roman province, and the find-spot coordinates place the record on Geodata. Persons named in an inscription are kept in the People register and linked back to it.

6 · Save and export

② Save to GitHub commits the record to records/<file>.xml in the public repository pleuston/simple-edep, served back as a static file; you can also Copy XML or Download and commit by hand. Every record is emitted with the EpiDoc RNG/Schematron processing instructions and follows the EpiDoc Guidelines. The Inscriptions list and Geodata map read data/records-index.json; after bulk edits, rebuild it with node scripts/build-index.js.

7 · GitHub sign-in

Signing in lets you commit records straight to GitHub under your own account. No password is stored — only a Personal Access Token (PAT) you generate once.

Create a token

  1. Go to GitHub → Settings → Developer settings → Personal access tokens → Tokens (classic).
  2. Give it any name (e.g. simple-edep). Set No expiration or a date you can remember.
  3. Tick the public_repo scope (write access to public repos). Use repo instead if your corpus is in a private repository.
  4. Click Generate token and copy it — GitHub shows it only once.

Sign in

  1. Open Login (or click Login in the top-right).
  2. Paste your token and click Sign in. The app calls GET /user on the GitHub API to verify the token and retrieve your username, display name, and avatar.
  3. You are returned to the page you came from. Your avatar and @username appear in the top-right corner on every page.
  4. Your identity and token are kept in browser storage only — they are never sent to any server other than api.github.com.

Returning visits

Your identity persists across browser restarts (localStorage). When you reopen the site after a break, the topbar shows a Welcome back, @username chip; click it to confirm or sign out without having to re-enter your token.

Sign out

Click your avatar / @username in the top-right. This ends the session immediately. Your saved token and identity are kept for the next visit — the Welcome-back chip will appear. To remove your token and identity completely (e.g. to switch accounts), click Sign out on the Login page.

Tip — token scope. If you are committing to a private fork or corpus repository, the public_repo scope is not enough. Generate a second token with the full repo scope for that repo, or use a fine-grained token scoped to only that repository.

8 · Submitting without a token

You can contribute records without a GitHub account. The editor button changes to ② Submit as guest when you are not signed in. Clicking it opens a modal with two routes:

RouteWhat happensResult
Submit anonymously The record is relayed through a small server-side proxy and committed to submissions/ in the project repository. A link to the new file on GitHub is shown. A maintainer can review and move it to records/.
Propose as pull request GitHub's new-file editor opens in a new tab with your XML pre-filled. You need a GitHub account but no token. You submit the PR yourself; GitHub shows the diff for review before merge.

Both routes write to submissions/, not directly to records/, so a maintainer can review before the record goes live. If the anonymous relay is not configured for this deployment, only the pull-request route is available.

Tip — large records. GitHub's URL length limit caps how much XML can be pre-filled for the PR route. Very long records (complex transcriptions, extensive apparatus) may be truncated. In that case, use Download to save the XML locally and attach it to a GitHub issue instead.

9 · Favorites and user collections

You can save any inscription to a personal list or a named collection — no account needed. Everything is stored in your browser only and is never uploaded anywhere.

Star an inscription

Every row in Inscriptions has a ★ button at the right. The same button appears in the header of the reading panel when you open a record. Click it to save the inscription to your Favorites.

Collections

Click the ★ button and then Save to… to open the collection popover. It shows your Favorites list and any named collections you have created. Tick or untick a collection to add or remove the inscription from it. You can save the same record to multiple collections at once.

  • New collection — type a name in the field at the bottom of the popover and press Enter (or click Add). The new collection is created and the record is saved into it immediately.
  • Close the popover by clicking anywhere outside it or pressing Esc.

Manage on the Favorites page

Open Favorites from the top navigation to see all saved inscriptions and your collections in one place.

  • Each collection has Rename and Delete buttons in its header. Deleting a collection removes the collection and its saved list but does not affect records in other collections or in Favorites.
  • Each saved row has its own ★ button — click it to remove the inscription from that collection without leaving the page.
  • Click an inscription title to open it in the reading panel.

Cross-tab sync

Favorites are stored in localStorage. If you have the site open in more than one tab, starring or removing an inscription in one tab is reflected immediately in the other.

Note — browser-local only. Favorites are not backed up to GitHub and are not shared between devices or browsers. If you clear your browser's site data, your collections will be lost. For a lasting record, use Copy XML or commit the records you care about to your own GitHub repository.
